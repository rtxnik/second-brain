# 36.01 Монолит

Монолитная архитектура - приложение как единый развёртываемый блок. Все компоненты тесно связаны, используют общую кодовую базу, единую БД и деплоятся вместе.

## Характеристики

**Единая кодовая база** - весь код в одном репозитории

**Общая база данных** - единое хранилище для всех компонентов

**Единое развёртывание** - один артефакт для деплоя

**Shared Memory** - компоненты общаются через вызовы функций

## Преимущества

| Преимущество        | Почему важно                      |
| ------------------- | --------------------------------- |
| Простота разработки | Всё в одном месте, легко дебажить |
| Простота деплоя     | Один артефакт                     |
| Производительность  | Нет сетевых вызовов               |
| Консистентность     | ACID транзакции в единой БД       |
| Низкий overhead     | Нет сетевой латентности           |

## Недостатки

| Недостаток      | Проблема                                |
| --------------- | --------------------------------------- |
| Масштабирование | Только целиком                          |
| Технологии      | Сложно использовать разные стеки        |
| Риски деплоя    | Ошибка в одном компоненте ломает всё    |
| Координация     | Большие команды сложно синхронизировать |
| Время сборки    | Растёт с кодовой базой                  |

## Когда выбирать

- Небольшие команды (до 8-10 разработчиков)
- Простые приложения с ограниченным функционалом
- Startup/MVP - быстрая валидация идей
- Строгие требования к консистентности данных

## Эволюция

Монолиты развиваются в [[36.02 Микросервисы]] через паттерны Strangler Fig или Database-per-Service при росте сложности.
